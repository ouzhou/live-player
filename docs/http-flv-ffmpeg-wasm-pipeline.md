# HTTP-FLV（H.264 + 音频）播放链路

面向：**网络状态良好**、**直播**、**FFmpeg 编译为 WASM** 做解码。不涉及弱网重连、花屏恢复等增强能力。

---

## 1. 端到端链路（总览）

```
HTTP 拉流（流式读取 body）
        ↓
FLV 解封装（分离视频轨 / 音频轨 + 时间戳）
        ↓
┌───────────────────────┬────────────────────────┐
│ 视频：H.264 NAL       │ 音频：常见为 AAC       │
│ + AVC 序列头(SPS/PPS) │ （FLV 内封装）         │
└───────────────────────┴────────────────────────┘
        ↓                         ↓
  FFmpeg WASM 解码           FFmpeg WASM 解码
        ↓                         ↓
  YUV / NV12 等              PCM（浮点或 S16 等）
        ↓                         ↓
  渲染（WebGL / Canvas / VideoFrame）
        ↓                         ↓
        └──────────┬──────────────┘
                   ↓
           音视频同步（PTS + 小缓冲队列）
                   ↓
           画面显示 + Web Audio 出声
```

---

## 2. 关键词简释

下面先给出**发送侧**（采集 → 上网）一条线的概念，便于把 **数据、H.264、FLV、流** 串起来；**播放侧**是逆过程：拉流 → 解封装 → 解码 → 显示/出声。

### 从数据到流（发送侧概念流程）

文字步骤（自上而下即一条完整链路）：

1. **原始数据**：摄像头/采集得到的图像采样（如 YUV）、麦克风 **PCM** 等未压缩数据。
2. **H.264 压缩（及音频编码）**：视频按 **H.264 / AVC** 压成码流（**NAL** 为主，**SPS/PPS** 进序列头）；音频常见再压成 **AAC**，与视频并行进入下一步。
3. **FLV 封装**：用 **FLV** 容器把已编码的音视频包打成带时间戳的 **Tag**（容器不管「怎么压像素」）。
4. **流**：通过 **HTTP-FLV** 等协议把 FLV 字节**持续发送**；对端边收边解，无需等整文件。

也可写成一行：**原始数据 → H.264 等压缩 → FLV 封装 → 网络流**。

**播放侧逆序**：`流` → 按 FLV 拆 Tag → 取出 H.264 / AAC 等负载 → **FFmpeg WASM 解码** → 像素与 PCM → 渲染与出声。

### 直播（FLV）里的时间戳 vs MP4 里的时间戳

两者都会用到 **PTS**（呈现时间戳）、**DTS**（解码时间戳）这类概念，但**语义和使用场景**不同。

**直播（如 FLV 推流 / HTTP-FLV 拉流）**

- 时间戳多表示**相对本场推流/会话**的时间（常用毫秒等），从推流开始**单调递增**，用于：解码顺序、音画对齐、低延迟调度。
- 流是**连续到达**的，播放器通常**不能**像文件那样任意 seek 到「中间某一秒」（除非业务做时移/回看，那是另一套协议与缓冲）。
- 断流重推后，时间戳**可能从较小值重新计数**，客户端要以「新的一段流」来衔接或重拉。

**MP4（点播文件）**

- 文件内部有完整**媒体时间轴**，时间戳在**文件头与 sample 表**里定义，通常可从 **0 覆盖到整片时长**，并支持 **seek** 到任意合法时刻。
- 编辑轨、存在 B 帧时，**DTS 与 PTS 可能不一致**（先解码、后呈现），但都在**固定文件结构**里可查。
- 更强调「**整段内容的进度条位置**」与随机访问，而不是「实时到达的增量」。

**对比小结**

| 维度     | 直播（FLV 等）                 | MP4（点播）                       |
| -------- | ------------------------------ | --------------------------------- |
| 时间轴   | 随推流持续延伸的**相对时间线** | 文件内**预先定义**的完整时间轴    |
| 典型用途 | 音画同步、低延迟播放           | 进度、seek、剪辑                  |
| 到达方式 | 字节**陆续到达**               | 文件**整体可读**（或 range 请求） |

实现播放器时：**直播侧**更关注「按到达顺序 + PTS 做同步与缓冲」；**点播 MP4** 更关注「时间基换算、seek 与 sample 索引」。底层解码器仍都按 **PTS/DTS** 工作，差异主要在**容器如何携带这些时间信息**以及**业务是实时流还是随机访问文件**。

## 3. FFmpeg WASM 在链中的职责

| 负责                                                        | 不负责                                                  |
| ----------------------------------------------------------- | ------------------------------------------------------- |
| 将 **压缩视频帧**（H.264）解码为 **原始像素格式**（如 YUV） | 从网络读流、HTTP 细节                                   |
| 将 **压缩音频帧**（如 AAC）解码为 **PCM**                   | FLV 容器解析（除非你在 WASM 内连 libavformat 一并使用） |
| （若集成 libavformat）也可在 WASM 内完成 demux + decode     | 浏览器里的显示与发声（需自行对接 Web API）              |

**串联方式二选一（必须明确）：**

1. **JS 侧 FLV demux**：主线程或 Worker 解析 FLV，按帧把 **NAL / AAC raw** 送入 FFmpeg WASM —— WASM 可只保留 decode，体积与接口相对简单。
2. **WASM 内 demux + decode**：把 **FLV 字节块** 喂给集成 **libavformat + libavcodec** 的 WASM —— 少写一层 FLV，但集成与产物体积通常更大。

## 4. 中间环节清单（易缺项）

| 环节          | 作用                                                                                | 说明                                                                          |
| ------------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **拉流**      | `fetch` 流式读 body，或 XHR，持续得到 `Uint8Array`                                  | 良好网络下仍需约定：流结束、关闭时的行为（可简化）                            |
| **FLV demux** | 解析 tag，得到 H.264 / AAC 及 **timestamp / compositionTime**                       | 浏览器里通常**不是**「FFmpeg 直接播 URL」，而是向内存中的 **FLV 数据** 解封装 |
| **送解码器**  | 按帧边界送入 WASM；视频需保证 **AVC sequence header（SPS/PPS）** 在可解码帧之前送达 | 缺序列头易导致无法解码或花屏                                                  |
| **视频输出**  | YUV → 屏幕                                                                          | 需 **渲染路径**：WebGL 着色器、Canvas、`VideoFrame` 等                        |
| **音频输出**  | PCM → 扬声器                                                                        | 需 **Web Audio**：`AudioContext`、队列、`AudioWorklet` / buffer               |
| **同步**      | 音画对齐                                                                            | 即使用小缓冲，也需按 **PTS**（及少量队列）调度，否则易漂移或爆音              |

## 5. 最小闭合条件

在「只考虑 H.264 + HTTP-FLV 直播 + FFmpeg WASM」前提下，除 WASM 本身外，还需同时具备：

1. **FLV 层**：独立 demux，或在 WASM 内用 libavformat。
2. **时间戳与缓冲**：解码前后队列 + PTS 对齐。
3. **两条输出**：**YUV（或兼容格式）→ 显示**，**PCM → Web Audio**。

满足以上三点，主链路即可跑通；重连、降缓冲策略、错误恢复等可作为后续增强。
