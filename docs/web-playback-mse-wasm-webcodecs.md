# Web 直播解码三种方案：MSE / WASM / WebCodecs

面向：**浏览器内**播放直播（典型如 HTTP-FLV、H.264）。说明各方案**是什么**、以及**数据如何一步步处理**（不解具体库名，只讲链路）。

---

## 总览对比

| 维度              | **MSE**                                                                                         | **FFmpeg WASM**                                                      | **WebCodecs**                                                                                    |
| ----------------- | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **是什么**        | 先把 **HTTP-FLV 解封装**，再**封装成 fMP4**，用浏览器 API **`MediaSource`** 喂给 `<video>` 硬解 | 在 WASM 里跑 **FFmpeg/libav**，**软解**，画面多画在 **Canvas/WebGL** | 浏览器 API：用 **`VideoDecoder`/`AudioDecoder`** 把**编码帧**解成 **`VideoFrame`/PCM**，自行渲染 |
| **解码谁做**      | **系统/浏览器内置**（硬解为主）                                                                 | **WASM 内软解**                                                      | **浏览器提供的解码器**（多为硬解，视 `isConfigSupported`）                                       |
| **典型体积/开销** | 无大块 WASM                                                                                     | **FFmpeg WASM 体积大、CPU 高**                                       | 无 giant FFmpeg，但要做帧调度                                                                    |

### 三种方案都要先解 FLV 吗？

**都要**——**没有任何一种**能让浏览器「直接吃整段 FLV 字节」就完成 MSE 或 WebCodecs 解码；**必须先解封装**，得到 **H.264 / AAC 负载 + 时间戳**（见下文统一章节）。

- **MSE / WebCodecs**：解 FLV 几乎都在 **JS 侧**完成——可以**自写解析**，更常见是**用现成库**（如 flv.js 一类）；**不是**必须手写每一字节，但**这一步不能省**。
- **FFmpeg WASM**：**二选一**——在 **JS 里解 FLV** 再喂解码器，或把 FLV 块交给 WASM 里的 **libavformat** 解；后者时 **JS 可以不写** FLV Tag 解析，但 **WASM 里仍在做 demux**。

所以：**「解 FLV」在三种方案里都存在**；区别只在 **由谁解（JS / 库 / libavformat）**，以及 **你是否亲自写解析代码**。

三种可以**组合**（例如 WASM 只负责 demux，解码走 WebCodecs），下文按**单一路径**说明。

---

## FLV 解封装之后：H.264 / AAC 是什么、时间戳怎么用

下面说明：**从 FLV 里拆出来的负载「长什么样」**，以及常说的 **「按 Tag / 按帧拆出 NAL、AAC，并带上时间戳」** 指什么。三种方案（MSE / WASM / WebCodecs）**共用同一套 demux 认知**，只是后面接法不同。

### FLV 里先拆到什么

HTTP-FLV 字节按 **FLV 文件/流格式** 解析后，得到一串 **Tag**（常见：**音频 Tag**、**视频 Tag** 交错排列）。每个 Tag 自带：

- **Tag 头**：类型（音频 / 视频）、**数据长度**、**时间戳**（毫秒级，还有扩展位）、等。
- **Tag 数据区**：即下面要说的 **H.264 负载** 或 **AAC 负载**（已是**压缩数据**，不是 YUV/PCM）。

也就是说：**解封装的第一步输出** = **「一类 Tag + 该 Tag 的 payload 二进制 + 该 Tag 的时间戳」**。

### 视频：H.264（FLV 里常标为 AVC）

视频 Tag 里，在 **CodecID = 7（H.264）** 时，数据区是 **AVC 视频包**，结构上大致是：

1. **AVCPacketType**（1 字节）
   - **0**：**AVC 序列头**（常称 **AVCDecoderConfigurationRecord**），里面是 **SPS/PPS** 等，解码器要靠它初始化；**不是**一帧画面。
   - **1**：**NAL 单元负载**（真正的一帧/一片里的编码数据），一般内含 **一个或多个 NAL**（FLV 里常见 **4 字节长度 + NAL 内容** 重复多组）。
   - **2**：序列结束等（直播里较少见）。

2. **CompositionTime**（3 字节）：**合成时间偏移**，和 Tag 上的时间戳一起，用来算**呈现顺序**上的时间（和「解码顺序」可能因 B 帧等不一致有关）。

3. **后续字节**：序列头或 NAL 数据本体。

所以：**「拆出来的 H.264 数据」** 要么是 **初始化用的序列头（含 SPS/PPS）**，要么是 **带 NAL 的视频包**；二者都是 **二进制块**，**不是**已经解码好的图像。

**「按帧」在视频侧通常指**：按 **AVCPacketType = 1** 的包，把里面的 **NAL** 取出来（可能一组长度前缀 NAL 对应**一帧**的一个 slice，或一帧多个 NAL），并带上 **该 Tag 的时间戳 + CompositionTime** 换算后的**展示时间**，再送给解码器或再封装。

### 音频：AAC

音频 Tag 里，**SoundFormat = 10（AAC）** 时，数据区大致是：

1. **AACPacketType**（1 字节）
   - **0**：**AudioSpecificConfig（ASC）**，描述采样率、声道、对象类型等，**AAC 解码器要先收到它**（或等价信息）。
   - **1**：**Raw AAC 帧数据**（常见是 **一帧 AAC 一 Tag** 的直播场景）。

2. **后续字节**：ASC 或 **一帧压缩后的 AAC 比特**。

所以：**「拆出来的 AAC 数据」** 要么是 **配置头**，要么是 **一帧一帧的 AAC 负载**（仍是压缩的，不是 PCM）。

### 时间戳：三个层次别混

| 名称                                 | 在哪         | 作用                                                                                                                         |
| ------------------------------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **Tag 时间戳**                       | FLV Tag 头   | 该 Tag 的基准时间（常用 **毫秒**），音/视频**各自 Tag 上都有**。                                                             |
| **CompositionTime**                  | AVC 视频包内 | 在 Tag 时间戳基础上的**额外偏移**，用于**正确显示顺序**（尤其有 B 帧时）。                                                   |
| **送给解码器 / fMP4 / Chunk 的时间** | 你算出来     | 把 Tag 时间戳（及视频的 CompositionTime）换算成 **PTS/DTS** 或 **微秒**，供同步、封装或 `EncodedVideoChunk.timestamp` 使用。 |

直播里常简化为：**按 Tag 顺序递增**，用 **Tag 时间戳（+ CompositionTime）** 做音画对齐。

### 小结一句

- **H.264 侧**：拆出来的是 **序列头（SPS/PPS 等）** 或 **NAL 二进制**（常带 **4 字节长度前缀**），配 **时间戳 + CompositionTime**。
- **AAC 侧**：拆出来的是 **ASC** 或 **逐帧 AAC 压缩数据**，配 **Tag 时间戳**。
- **「按帧拆」**：在 FLV 里多数情况下就是 **按 Tag 拆**（视频一个 NAL 包常对应一帧或一个访问单元；音频常 **一 Tag 一 AAC 帧**），再把**时间戳**绑在同一单元上交给下游（fMP4、WASM、WebCodecs）。

## 1. MSE（Media Source Extensions）

### 一句话（HTTP-FLV 场景）

**是的**：常见做法就是 **把 HTTP-FLV 解封装（demux）**，再 **重新封装（remux）成 fMP4**，然后才交给 MSE；**不是**把 FLV 原样塞进 `SourceBuffer`。

### 是什么

- **`MediaSource`** 创建 `<video>` 可用的媒体源；**`SourceBuffer`** 吃的是 **fMP4 / CMAF 这类片段**，**不直接吃 FLV 容器**。
- 浏览器在收到 fMP4 后，在内部完成：**解封装（对片段）→ 解码 → 输出到 `<video>`**。
- 所以：**FLV → fMP4 的转换发生在你的 JS（或库）里**，发生在调用 MSE 之前。

### 数据怎么处理（HTTP-FLV + H.264）

1. **HTTP 拉流**：持续读到 **FLV 字节**。
2. **解封装**：解析 FLV，拆出 **H.264 / AAC** 等**压缩负载**和时间戳。
3. **再封装成 fMP4**：把上述负载打成 **fragmented MP4** 小片段（mime 与 `codec` 要和 `MediaSource.isTypeSupported` 一致）。
4. **append**：`sourceBuffer.appendBuffer(fMP4 片段)`，按时间顺序追加。
5. **播放**：`<video>` 内硬解、显示、出声；业务侧一般**拿不到逐帧 YUV**。

**要点**：MSE 这条线的核心是 **「FLV demux → fMP4 mux → MSE」**；裸 H.264 NAL 也不能随便 append，要装进 **浏览器认的 MP4 轨**里。

## 2. FFmpeg WASM

### 是什么

- 用 **Emscripten** 把 **FFmpeg（或 libavcodec/libavformat）** 编译成 **`.wasm`**，在 **Worker** 里跑，避免卡主线程。
- **解码在 WASM 内完成**（软解），输出多为 **YUV 视频帧 + PCM 音频**。
- 画面需自己 **画到 Canvas / WebGL**；声音 **喂给 Web Audio**。

### 数据怎么处理

1. **HTTP 拉流**：同上，得到 **FLV 字节流**。
2. **解封装**（二选一，可只在 JS 做）
   - **JS 侧（常见）**：自己解析 **HTTP-FLV**，按帧拆出 **NAL / AAC** 与时间戳，再交给 Worker 里的 WASM；WASM 里往往**只编 libavcodec**，**不带 libavformat**，**体积更小**。
   - **WASM 内**：把 **FLV 字节块** 直接喂给 **libavformat**，在 WASM 里 demux + decode，集成简单但 **WASM 更大**。
3. **解码**：**libavcodec** 解 H.264/AAC → **YUV**（如 YUV420p、NV12）+ **PCM**（若 demux 在 JS，需保证送入格式与解码器约定一致，如 Annex-B / AVCC）。
4. **渲染与出声**：YUV → 着色器/WebGL 或转 RGB 画 Canvas；PCM → `AudioContext`；用 **PTS** 做音画同步。

**要点**：路径与桌面播放器类似，**数据全链路可控**；代价是 **包体、CPU、内存**。**「JS 解封装 → WASM 只解码」** 完全可行，也是为减小 WASM 的常用做法。

## 3. WebCodecs

### 是什么

- **`VideoDecoder`**：输入 **编码视频帧**（`EncodedVideoChunk`），输出 **`VideoFrame`**（可画 Canvas、`copyTo` 等）。
- **`AudioDecoder`**：输入 **编码音频帧**（`EncodedAudioChunk`），输出 **AudioData**（可转 PCM 进 Web Audio）。
- 解码由**浏览器实现**，通常**硬解**；需 **`VideoDecoder.isConfigSupported()`** 探测 **codec 字符串**（如 `avc1.xxx`）。

### 数据怎么处理

1. **HTTP 拉流**：同上，得到 **FLV 字节**。
2. **解封装（必做）**：解析 FLV，得到 **带时间戳的 H.264 NAL / AAC 帧**；必要时做 **AVCC / Annex-B** 等与 `VideoDecoder` 配置一致。
3. **配置解码器**：`new VideoDecoder({ output, error })`，`configure({ codec: 'avc1....', description: ... })`（含 SPS/PPS 等）。
4. **喂帧**：把每帧打成 **`EncodedVideoChunk`**，`decoder.decode(chunk)`；在 **`output` 回调**里拿到 **`VideoFrame`**，**`timestamp`** 用于同步。
5. **音频**：`AudioDecoder` 同理，输出接 Web Audio。
6. **渲染**：`VideoFrame` → **`canvasContext.drawImage(videoFrame)`** 或 **`WebGL`/`copyTo`**。

**要点**：**不解 FLV 协议本身**——只认**已拆好的编码帧**；FLV 仍要在前一步处理掉。

## 三种路径的数据形态小结

| 阶段                                                   | MSE                                                                | FFmpeg WASM                                        | WebCodecs                                                                           |
| ------------------------------------------------------ | ------------------------------------------------------------------ | -------------------------------------------------- | ----------------------------------------------------------------------------------- |
| 网络 → 内存                                            | FLV 字节                                                           | FLV 字节                                           | FLV 字节                                                                            |
| 解封装后                                               | **FLV demux** → NAL/AAC + 时间戳 → **remux 成 fMP4**               | NAL/AAC 或交给 **libavformat**                     | NAL/AAC + 时间戳 → **Chunk**                                                        |
| 解码后                                                 | **在 `<video>` 内**（不暴露 YUV）                                  | **YUV + PCM**                                      | **`VideoFrame` + AudioData**                                                        |
| 显示/出声                                              | `<video>`                                                          | Canvas/WebGL + Web Audio                           | Canvas + Web Audio                                                                  |
| **延迟倾向**（同等业务设定下**大致**对比，非固定毫秒） | **中等**：多一次 **FLV→fMP4**，`SourceBuffer` 往往要带**小段缓冲** | **易偏高**：**软解**占 CPU、链路里常再加缓冲防卡顿 | **可做得更低**：少一次 remux、多走**硬解**、**帧级**调度；仍受 demux 与队列策略影响 |

### 编码格式：通常都支持的 vs 设备/环境常不支持的

下面按三种路径分开写。**「通常都能用」**指桌面 Chrome / Edge / Firefox、移动端常见版本里**大概率**能播的；**「常受限」**指大量设备没有硬解或浏览器直接不支持，**必须探测或降级**。最终以 **`MediaSource.isTypeSupported`**、**WASM 编译列表**、**`VideoDecoder.isConfigSupported` / `AudioDecoder.isConfigSupported`** 为准。

#### MSE（`SourceBuffer` + 浏览器内置解码）

| 类型     | 通常广泛支持                                                                                                                                              | 常受限 / 需单独探测                                                                                                                                                                     |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **视频** | **H.264（AVC）**：Web 直播事实标准，**绝大多数**环境可播。**VP8**：常见。**VP9**：Chrome / Firefox / Android 系多数可；**旧 iOS Safari** 等场景可能不行。 | **HEVC（H.265）**：**Windows 上 Chrome 往往不支持**；**Safari / 部分 Apple、部分高端 Android** 上相对多。**AV1**：需较新 **Chrome / Edge / Firefox**；**老机型、部分 iOS** 可能不支持。 |
| **音频** | **AAC（LC）** 与 fMP4 搭配**最常见**。**MP3** 在部分 `audio/mpeg` 场景可用。                                                                              | **HE-AAC v2** 等变体、**多声道配置**要以 `isTypeSupported` 实测。**Opus** 多走 WebM / 特定封装，**进 MSE 的 fMP4 轨**里是否支持看 mime。                                                |

#### FFmpeg WASM（软解，能否解 = 编进 WASM 了没有）

| 类型     | 通常广泛支持（只要编译打开对应 decoder）                                                                                               | 常受限 / 注意                                                                                                                    |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **视频** | **H.264**：软解成熟。**HEVC、VP8、VP9、AV1**：**不依赖**浏览器是否硬解，**只要能编进 libavcodec 就能解**（**MPEG-4 Part 2** 等同理）。 | **代价**：**CPU、内存、耗电**；**4K / HEVC / AV1** 在**中低端手机**上可能卡顿。**版权/许可**：部分编解码器集成时要注意项目许可。 |
| **音频** | **AAC、MP3、Opus、Vorbis** 等，按编译选项。                                                                                            | 与浏览器无关，但 **WASM 体积**会随解码器数量增加。                                                                               |

#### WebCodecs（`VideoDecoder` / `AudioDecoder`）

| 类型     | 通常广泛支持                                                                                                                                                                         | 常受限 / 需 `isConfigSupported`                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **视频** | **H.264（avc1）**：**多数**现代浏览器硬解路径可用。**VP8、VP9**：Chrome / Firefox / Edge 桌面与多数 Android **常见**。**AV1**：**较新桌面 Chrome / Edge** 常见；移动端看机型与版本。 | **HEVC（hev1/hvc1）**：**大量 Windows / Android Chrome 不支持**；**Safari / Apple 设备**上相对常见。**AV1**：**旧浏览器、部分 iOS** 可能没有。 |
| **音频** | **AAC（mp4a）**、**Opus**：在支持的浏览器里**常见**。                                                                                                                                | **采样率 / 声道 / profile** 组合要逐个探测；不支持的配置会 `configure` 失败。                                                                  |

---

## 选用提示（极简）

- **H.264 为主、要省事、省电**：优先 **MSE（常配合 FLV→fMP4）**。
- **要逐帧像素、或多格式软解统一**：**FFmpeg WASM**（或 WASM 只 demux + 下面一种解码）。
- **要低延迟、硬解、可控队列**：在支持的前提下用 **WebCodecs**，前面仍要 **demux**。
